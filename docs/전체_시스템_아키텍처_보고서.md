## 1. 프로젝트 개요

본 프로젝트는 코딩테스트 학습자를 위한 **AI 기반 알고리즘 문제 풀이 지원 시스템**이다.

사용자가 알고리즘 문제를 입력하면 시스템이 다음을 자동으로 수행한다.

```text
1. 문제 분석 및 풀이 전략 수립
2. 파이썬 코드 생성 및 실행 검증
3. 오류 발생 시 자동 디버깅
4. 관련 알고리즘 개념 문서 검색 (RAG)
5. 접근법·알고리즘 설명·코드 해설이 포함된 최종 답변 생성
```

| 파트 | 담당자 | 주요 작업 |

## 4. 레이어별 구조 설명

### 4-1. API 레이어

FastAPI로 구성된 진입점이다. 사용자 요청을 받아 Agent에 전달하고 결과를 반환한다.

```text
app/main.py
  ├── health_router  (/health/*)
  └── rag_router     (/rag/*)
```

### 4-2. Agent 레이어

LangGraph StateGraph로 구현된 멀티스텝 에이전트이다. 문제 풀이 전 과정을 오케스트레이션한다.

```text
app/agent/solve_agent.py
  ├── SolveState       (공유 상태 객체)
  ├── 7개 노드          (think, generate_code, execute, debug,
  │                     algorithm_query, retrieve, final_answer)
  └── 2개 조건부 엣지   (execution_router, retry_router)
```

### 4-3. LLM 레이어

각 노드에서 사용하는 언어 모델을 제공한다.

```text
app/llm/
  ├── openai_model.py   GPT-4o-mini (temperature=0)
  ├── qwen_model.py     Qwen3-0.6B + LoRA 어댑터 (현재 비활성화)
  └── prompts.py        5개 프롬프트 템플릿
```

### 4-4. MCP 레이어

코드 실행 도구를 제공한다.

```text
app/mcp/
  └── tools.py          execute_python (FastMCP 기반)
```

현재는 Agent에서 직접 함수를 임포트하여 사용한다. MCP 서버 연동은 추후 추가 예정이다.

### 4-5. RAG 레이어

알고리즘 개념 문서를 검색하여 LLM 컨텍스트를 보강한다.

```text
app/rag/
  ├── retrieval_v1.py       Retrieval V1 (핵심 검색 로직)
  ├── retriever.py          기본 retriever 래퍼
  ├── vector_store.py       ChromaDB 연결
  ├── embeddings.py         OpenAI text-embedding-3-small
  ├── splitter.py           Adaptive Chunking
  ├── loader.py             문서 로더
  ├── ingest.py             데이터 수집 파이프라인
  └── konlpy_preprocessing.py  TF-IDF baseline용 전처리
```

## 5. 컴포넌트 연결 관계

```text
health_router
  └── solve_problem()
        └── solve_graph.invoke()          ← LangGraph
              ├── think_node
              │     └── get_qwen_model()  ← LLM (현재 OpenAI)
              ├── generate_code_node
              │     └── get_openai_model()
              ├── execute_node
              │     └── execute_python()  ← MCP tools
              ├── debug_node
              │     └── get_openai_model()
              ├── algorithm_query_node
              │     └── get_qwen_model()  ← LLM (현재 OpenAI)
              ├── retrieve_node
              │     └── get_retriever()
              │           └── get_vector_store()  ← ChromaDB
              └── final_answer_node
                    └── get_openai_model()

rag_router
  └── save_document()
        └── get_vector_store()            ← ChromaDB
```

## 6. API 엔드포인트

| 메서드 | 경로 | 설명 |
|--------|------|------|
| GET | `/health` | 서버 생존 확인 |


```json
{
  "answer": "1. 문제 접근법\n해시맵을 사용하여...\n2. 핵심 알고리즘 설명\n...\n5. 최종 정답 코드\n..."
}
```

## 8. 기술 스택

| 분류 | 기술 | 용도 |
|------|------|------|
| 서버 | FastAPI + Uvicorn | REST API 서버 |
| 에이전트 | LangGraph | 멀티스텝 에이전트 오케스트레이션 |
| LLM (추론) | OpenAI GPT-4o-mini | 코드 생성, 디버깅, 최종 답변 |
| LLM (로컬) | Qwen3-0.6B + PEFT/LoRA | 사고 과정, 쿼리 생성 (현재 비활성화) |
| 임베딩 | OpenAI text-embedding-3-small | 문서 및 쿼리 벡터화 |
| 벡터DB | ChromaDB | 알고리즘 문서 저장 및 검색 |
| RAG | LangChain + Retrieval V1 | 문서 검색 및 컨텍스트 보강 |
| 코드 실행 | FastMCP + exec() | Python 코드 샌드박스 실행 |
| 청킹 | Adaptive Semantic Chunking | 문서 분할 |
| 전처리 | KoNLPy (Okt) | 한국어 형태소 분석 |
| 환경 | conda (skn3rd) | 패키지 관리 |

## 9. 데이터 흐름 상세

### 9-1. 데이터 구축 파이프라인 (사전 작업)

```text
알고리즘 학습 문서 (.md)
    ↓
run_ingest.py 실행
    ↓
splitter.py — 포맷 감지 (A/B/C) + Adaptive Chunking
    ↓
embeddings.py — text-embedding-3-small 벡터화
    ↓
vector_store.py — ChromaDB 저장
    ↓
chroma_db/ (로컬 디렉토리)
```

구축 결과:

```text
문서 수:  201개
청크 수: 1686개
알고리즘 키: 110종
```

### 9-2. 런타임 요청 처리 파이프라인

```text
사용자 요청: "두 수의 합이 target인 인덱스를 찾아라."
    ↓
[1] think_node
    LLM → "해시맵 사용. O(N). 보수를 미리 저장하며 순회."
    ↓
[2] generate_code_node
    LLM → code / test_input / expected_output (JSON)
    ↓
[3] execute_node
    exec(code) → execution_result
    ↓
[4] execution_router
    result == expected? → YES: [5], NO: debug loop
    ↓
[5] algorithm_query_node
    LLM → "해시, 딕셔너리"
    ↓
[6] retrieve_node
    ChromaDB similarity search (Top-5)
    → 해시맵 개념 문서 5개
    ↓
[7] final_answer_node
    LLM(문제 + 사고과정 + 코드 + 개념문서) → 최종 답변
    ↓
사용자 응답 반환
```

## 10. 파트별 책임 경계

| 경계 | 내용 |
|------|------|
| RAG ↔ Agent | `get_retriever()`를 통해 연결. RAG 내부 로직(Retrieval V1)은 Agent가 알 필요 없음 |
| Agent ↔ LLM | `get_openai_model()`, `get_qwen_model()` 래퍼로 추상화. 모델 교체 시 래퍼만 수정 |
| Agent ↔ MCP | 현재 직접 임포트. 추후 MCP 클라이언트로 전환 예정 |
| API ↔ Agent | `solve_graph.invoke({"problem": request})`로 단일 진입점 유지 |
| 모델학습 ↔ 추론 | `adapters/` 디렉토리의 LoRA 어댑터를 `qwen_model.py`에서 로드. 학습과 추론 분리 |
